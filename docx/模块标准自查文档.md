# 模块标准自查文档

> 以模块一「HTML 基础」修复后的质量基线为基准，抽象出可复用的模块质量标准，供其它模块在新建/重构时对照自查，确保全站学习体验的一致性与有效性。

---

## 一、文档目的

本文件解决一个具体问题：**如何判断一个学习模块是否"达标"？**

模块一此前存在三类典型缺陷——评估环节形同虚设、知识呈现静态化、内容组织学科导向。经修复后，模块一在「以测促学 / 动态反馈 / 任务导向」三个维度上建立了可参照的基线。本文件把该基线抽象成 6 个自查维度 + 1 张勾选清单，使其它模块无需重复摸索即可对齐质量。

**适用对象**：所有 `src/modules/*.ts` 学习模块（模块二 CSS、模块三 JS …… 及后续新增模块）。

**使用方式**：模块开发/重构完成后，逐项对照第三章维度与第四章清单打勾；任一维度不达标即视为该模块未完成质量门禁。

## 三、自查维度

每个维度包含：**① 标准要求 → ② 模块一实例 → ③ 检查方法 → ④ 常见缺陷**。

### 维度 A：知识点结构与编排

**① 标准要求**

- A1. `points` 数组的 `order` 字段从 1 连续递增至 N，**无重复、无跳号、无小数**。
- A2. 每个知识点具备 `title` / `difficulty`(1–5) / `blocks` 三要素。
- A3. 难度分布合理：以 1–3 为主，进阶内容(4–5)不超过总量的 30%。
- A4. 新增/重构的知识点显式标注 `isNew: true`，便于学习者识别更新。
- A5. `blocks` 内 `id` 遵循 `p{order}-{n}` 命名规范，全模块唯一。

**② 模块一实例**

26 个知识点 order 1–26 严格连续；KP22/KP23 实战项目标注 `isNew: true`；难度集中在 1–2，仅个别 API 章节为 3。

**③ 检查方法**

```bash
# 提取所有 order 值，人工核对连续性
# （用 IDE 正则搜索 ^\s+order:\s+(\d+),）
```

或在模块测试中断言：

```ts
it('知识点序号应从 1 到 N 连续', () => {
  const orders = someModule.points.map((p) => p.order)
  for (let i = 0; i < orders.length; i++) {
    expect(orders[i]).toBe(i + 1)
  }
})
```

**④ 常见缺陷**

- 插入新知识点后未重排后续 order，导致跳号/重复（模块一曾出现 order 21 / 21.5 / 22 冲突）。
- 用小数 order（如 21.5）作为临时插入，破坏连续性约定。

---

### 维度 B：评估环节有效性（以测促学）

**① 标准要求**

- B1. **面试题 ≥ 12 题**，覆盖该模块所有核心考点，无高频考点遗漏。
- B2. **小测题 ≥ 10 题**，且分布在模块末尾的独立 QuizCard 知识点。
- B3. 题目具备**梯度**，能区分"记忆 / 理解 / 应用"三个层次：
  - 记忆层：直问定义、标签类型；
  - 理解层：概念辨析、原理说明；
  - 应用层：场景判断、代码选型、排错。
- B4. 面试题至少包含 **2 道场景题**（"遇到 X 问题你怎么排查/重构"）与 **2 道对比题**（易混淆点 A vs B）。
- B5. 每道题的 `explanation` / `content` 不仅给结论，还解释"为什么"，指向易错点。
- B6. 考点覆盖率自查：列出模块所有核心知识点，逐个标注是否有对应题目，无盲区。

**② 模块一实例**

- 面试题 16 题：Q14 场景题（页面卡顿排查）、Q15 场景题（div 导航重构）、Q8 对比题（标准/IE 盒模型）、Q16 对比题（`<b>`/`<strong>` 等）。
- 小测题 15 题，题干前缀标注【理解】【应用】【对比】【场景】【综合】。
- 考点覆盖：文档流、盒模型、meta、viewport、语义化、defer/async、无障碍 POUR、ARIA、表单验证、srcset、空元素等均有对应题目。

**③ 检查方法**

```bash
# 统计面试题数量（匹配 Q{n}: 前缀）
# 统计小测题数量（匹配 ^\s+question:）
# 人工核对：是否存在"学了但没考"的知识点
```

**④ 常见缺陷**

- 题目全是直问定义，无法区分背书与理解（缺应用层）。
- 高频考点（如模块一的"文档流""表单验证"）无对应题目。
- 题目集中在模块末尾，章节学完无即时检验（理想态：每章末可加微测验，至少模块级总测验要全覆盖）。

### 维度 C：动态反馈闭环（沙盒断言）

**① 标准要求**

- C1. `SandboxData` 类型支持 `checks?: SandboxCheck[]` 字段（已由 types.ts 全局提供，各模块直接使用）。
- C2. `Sandbox.tsx` 渲染"任务检查清单"面板，对学习者代码**实时**跑断言并显示通过/未通过 + 中文教学提示（已由组件全局提供）。
- C3. 模块内**任务型练习沙盒**（尤其是实战项目）必须配置 `checks`，至少 5 条。**自由探索型沙盒**（hint 为"修改 X 观察 Y"的演练场）可不配 checks，因其教学目的是自由试错而非任务达标。区分依据：沙盒是否设定了明确的"完成目标"。
- C4. 每条 check 三要素齐全：`description`（任务要求）/ `pattern`（正则）/ `hint`（未通过时的教学引导）。
- C5. `pattern` 必须在沙盒中可正确编译（避免正则语法错误导致永远不通过）。
- C6. 断言不只是"语法存在性"检查，应体现**教学意图**（如"必须写 alt"对应无障碍教学点）。
- C7. 全部通过时显示完成态，给学习者正反馈。

**② 模块一实例**

- KP22 博客文章页沙盒：9 条 checks（charset / viewport / header / nav / main / article / img+alt / aside / footer）。
- KP23 注册表单沙盒：8 条 checks（form / text+required / email+required / password+minlength / select / button[type=submit] / label[for] / aria-required）。
- 每条 hint 指出"该怎么做 + 为什么"，例如 img+alt 的 hint 解释"无 alt 的图片对视障用户不存在，违反 WCAG 可感知原则"。

**③ 检查方法**

```bash
# 统计 checks 数量（匹配 ^\s+pattern:）
# 人工核对：每个配置了 checks 的沙盒，其 pattern 是否能在预期代码上通过、在错误代码上失败
```

**类型字段速查**：

```ts
export interface SandboxCheck {
  description: string  // 任务要求（展示给学习者）
  pattern: string      // 正则字符串，命中即通过
  flags?: string       // 默认 'i'
  hint: string         // 未通过时的教学提示
}
```

**④ 常见缺陷**

- 沙盒只支持"编辑-预览"，写错只能看浏览器原生报错，无教学反馈（模块一修复前的状态）。
- `pattern` 写得过严或过松：过严导致正确写法被判失败，过松导致随便写都通过。
- 只有语法检查没有语义检查（如只查"有没有 `<img>`"而不查"有没有 alt"）。

### 维度 D：任务导向内容组织（综合实战项目）

**① 标准要求**

- D1. 模块内至少 **1 个**（建议 2 个）综合实战知识点，放在该模块核心章节学完之后。
- D2. 实战项目必须**贯穿多个章节**的知识点，扮演"拼装图纸"角色，而非单点练习。
- D3. 每个实战项目配 `checks` 断言沙盒（接入维度 C）。
- D4. 沙盒提供**初始骨架代码**（带注释占位），降低冷启动门槛。
- D5. 项目前后有 `callout` 说明"为什么这个练习重要"与"实战后的反思"，连接工程思维。
- D6. 项目命名体现**真实开发场景**（如"搭建博客文章页""带验证的注册表单"），而非"练习 1"。

**② 模块一实例**

- KP22「搭建博客文章页」：串联 head 元信息 + 语义化骨架 + 多媒体无障碍，9 条断言，含 `callout variant="tip"` 说明练习价值。
- KP23「带验证注册表单」：串联表单控件 + 原生验证 + 无障碍 label 关联，8 条断言，含 `callout variant="warning"` 反思"前端验证的边界"（衔接后端校验）。
- 两者均提供带 `<!-- -->` 占位注释的初始骨架。

**③ 检查方法**

```bash
# 搜索实战项目知识点（title 含"实战"/"综合"）
# 核对每个实战沙盒是否配置 checks、是否有初始骨架、是否有 callout
```

**④ 常见缺陷**

- 知识按学科分类（结构→语义→多媒体→表单→API），学完仍不会"从零搭页面"。
- 没有贯穿多章的渐进式项目，知识点是离散积木缺拼装图纸。
- 实战项目变成"照抄示例"，无断言引导，无法定位薄弱环节。

---

### 维度 E：辅助学习资源

**① 标准要求**

- E1. 模块末尾含 1 个**知识点速查表**（`visualizationType: 'cheatsheet'` 或等价表格），浓缩核心 API/标签/属性。
- E2. 关键概念处使用 `callout`（`tip` / `warning` / `info`）强调易错点、最佳实践、边界条件。
- E3. 首个知识点有 `lead: true` 的引导段落，说明本章学什么、为什么学。
- E4. 代码示例遵守项目设计规范（canvas-mid 代码区、hairline 边框、等宽字体）。

**② 模块一实例**

- KP25 知识点速查表。
- 多处 callout：如表单章节的"无障碍表单要点"(tip)、KP23 的"前端验证边界"(warning)。
- 每章首段 `lead: true` 引导。

**③ 检查方法**

```bash
# 搜索 callout 使用密度（匹配 type: 'callout'）
# 确认存在 cheatsheet / 速查表知识点
```

**④ 常见缺陷**

- 全文平铺直叙，无重点强调，学习者抓不住易错点。
- 缺速查表，复习时无法快速定位。

---

### 维度 F：类型与工程质量

**① 标准要求**

- F1. `npx tsc --noEmit` 通过（exit 0），无类型错误。
- F2. 存在对应模块测试 `src/test/{module}.test.ts`，且全绿。
- F3. 测试覆盖：知识点数量、order 连续性、每个知识点三要素、可视化组件元数据。
- F4. 模块数据严格符合 `ModuleData` 类型（`points` / `blocks` / `data` 结构正确）。
- F5. 新增 `visualizationType` 时同步登记到 `visualizationMeta`（避免 types.test.ts 漂移）。
- F6. 不引入与模块无关的全局类型变更（避免污染其它模块）。

**② 模块一实例**

- `tsc --noEmit` exit 0。
- `html-fundamentals.test.ts` 14/14 通过（含 26 知识点、order 连续、三要素校验）。
- 注：`types.test.ts` 的 `visualizationMeta` (105 vs 173) 失败为**预先存在**的全局漂移，与模块一无关，需单独治理。

**③ 检查方法**

```bash
npx tsc --noEmit
npx vitest run src/test/{module}.test.ts --reporter=dot
```

**④ 常见缺陷**

- 新增知识点后忘更新测试中的 `toHaveLength(N)`，导致测试红。
- 新增可视化类型未登记 `visualizationMeta`，触发 types.test.ts 失败。
- order 用小数临时插入导致连续性测试失败。

## 五、应用流程（如何把本标准用到新模块）

1. **新建模块骨架**：复制 `html-fundamentals.ts` 的结构（顶部 import、`ModuleData` 形状），改 `number/title/slug/stage/icon`。
2. **编写知识点**：按 `order` 1..N 顺序编写，遵循 A 系列标准。
3. **穿插实战项目**：在核心章节后插入综合实战 KP，配置 `checks` 沙盒（D + C 标准）。
4. **编写评估题**：先列出全部核心考点，再为每个考点配题，确保 B6 无盲区；题目前缀标注梯度。
5. **补速查表与 callout**：模块末尾加速查表，关键处加 callout（E 标准）。
6. **编写模块测试**：参照 `html-fundamentals.test.ts`，覆盖数量/order 连续/三要素。
7. **跑质量门禁**：
   ```bash
   npx tsc --noEmit
   npx vitest run src/test/{module}.test.ts --reporter=dot
   ```
8. **对照第四章清单逐项打勾**，未勾项回到对应维度修复。
9. **登记可视化类型**：若引入新 `visualizationType`，同步更新 `visualizationMeta` 与 `types.test.ts`。

### 6.3 关键决策记录

- **自由探索型沙盒不强制配 checks**：模块二原有 3 个沙盒（Flexbox/动画/CSS 变量）hint 为"修改 X 观察 Y 变化"，属自由探索演练场而非任务型练习。若强加 checks 会改变其"自由试错"的教学定位。据此将标准 C2 细化为"任务型沙盒必须配 checks，自由探索型可不配"，并以"是否设定明确完成目标"作为区分依据。此修订已回写到第三章维度 C 与第四章清单。
- **实战项目顺序**：遵循模块一模式（核心 KPs → 实战项目 → 面试题 → 速查表 → 小测验），确保小测验作为收尾评估。
